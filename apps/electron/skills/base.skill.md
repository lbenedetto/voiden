# Voiden .void File Format

## Overview

Voiden uses `.void` files for API testing. Each file uses a **hybrid format** combining:
- YAML frontmatter for metadata
- Markdown content for documentation
- Custom `void` code blocks for structured request/response data

## File Structure

```markdown
---
version: 0.20.1
generatedBy: Voiden app
note: This file is auto-generated by the Voiden app
generatedAt: 2025-01-15T10:30:00.000Z
---

# API Documentation Title

Description and test cases in Markdown...

```void
---
type: request
attrs:
  uid: 05119029-b827-45fe-a2cc-55a5f0ba6609
content:
  - type: method
    attrs:
      uid: 35d05fd6-1e12-4f6a-97d5-5c66d59bc61f
      method: GET
      visible: true
    content: GET
  - type: url
    attrs:
      uid: f84b83da-7fe7-415f-baa1-b045adfd7438
    content: https://api.example.com/users
---
```
```

## Frontmatter Fields

| Field | Value |
|-------|-------|
| `version` | Current Voiden version (e.g. `0.20.1`) |
| `generatedBy` | Always `"Voiden app"` |
| `note` | Always `"This file is auto-generated by the Voiden app"` |
| `generatedAt` | ISO 8601 timestamp |

## Void Block Structure

All blocks use triple-backtick `void` fences:

```yaml
---
type: [block-type]
attrs:
  uid: [UUID v4]
  [other-attrs...]
content:
  [child nodes or string value]
---
```

## Rules

- **UIDs**: Always UUID v4 — `crypto.randomUUID()` or `uuidgen`
- **Variables**: Use `{{VARIABLE_NAME}}` for environment variables
- **Indentation**: 2 spaces for YAML and JSON
- **Strings**: Quote values containing special characters, colons, or spaces
- **Timestamps**: ISO 8601 format (`2025-01-15T10:30:00.000Z`)
- **Block order inside `request`**: method → url → auth → headers-table → query-table → path-table → body → assertions → scripts

## Workflows

### Convert curl to .void

1. Parse HTTP method from `-X` flag (default GET)
2. Extract URL (from `--url`, bare argument)
3. Map `-H "Key: Value"` → `headers-table` rows
4. Map `?key=value` query params in URL → `query-table` rows
5. Map `-d '{...}'` → `json_body` (or `text_body` for non-JSON)
6. Map `-u user:pass` → `auth` block with `authType: basic`
7. Map `-H "Authorization: Bearer ..."` → `auth` block with `authType: bearer`
8. Generate fresh UUIDs for all `uid` attrs
9. Write ISO timestamp for `generatedAt`

### Generate from OpenAPI/Swagger spec

1. Identify all `paths` + HTTP methods combinations
2. Group related endpoints into one `.void` file by resource (e.g. `users.void`)
3. For each endpoint: create a `request` block with `method` + `url`
4. Add `path-table` for `{param}` segments
5. Add `query-table` for query parameters
6. Add `json_body` from the request body schema
7. Add `auth` matching the spec's `securitySchemes`
8. Write markdown documentation above each request block

## Importing Blocks from Other Files

Individual blocks can be imported from any other `.void` file using a `linkedBlock`. This lets you reuse headers, auth, query params, bodies, etc. across requests without duplicating them.

### linkedBlock — Import a Specific Block

```yaml
---
type: linkedBlock
attrs:
  uid: 1ac1db1b-26fa-486a-9d74-1f8db6b506be
  blockUid: 825cb614-99af-4d19-ac1e-7af0dcb2acf5
  originalFile: /blocks/headers.void
---
```

| Attr | Description |
|------|-------------|
| `uid` | Fresh UUID v4 generated for this `linkedBlock` instance |
| `blockUid` | The `uid` of the specific block being imported from the target file |
| `originalFile` | Absolute path to the target file from the **project root** (e.g. `/blocks/headers.void`) |

### How to Generate a linkedBlock

When asked to import a block from another file:

1. **Read the target `.void` file** to find the block the user wants (headers-table, auth, json_body, etc.)
2. **Copy its `uid`** — that becomes `blockUid`
3. **Generate a fresh UUID** for the `linkedBlock`'s own `uid`
4. **Set `originalFile`** to the path of that file from the project root

### Override Logic

**Local blocks always win over imported ones.** When the same key appears in both an imported block and a local block of the same type, the local value takes precedence. You can import a shared headers-table and add or override individual rows with a local headers-table in the same request.

### Examples

**Import headers from `/shared/base-headers.void`** (which contains a headers-table with uid `825cb614-...`):

```yaml
---
type: linkedBlock
attrs:
  uid: 1ac1db1b-26fa-486a-9d74-1f8db6b506be
  blockUid: 825cb614-99af-4d19-ac1e-7af0dcb2acf5
  originalFile: /shared/base-headers.void
---
```

**Import auth from `/auth/login.void`** (which contains an auth block with uid `f3a2b1c0-...`):

```yaml
---
type: linkedBlock
attrs:
  uid: 9d3e7f12-aa45-4b89-cd01-ef2345678901
  blockUid: f3a2b1c0-1234-5678-abcd-ef0123456789
  originalFile: /auth/login.void
---
```

**Import a JSON body from `/payloads/create-user.void`** (json_body uid `cc112233-...`):

```yaml
---
type: linkedBlock
attrs:
  uid: 7b6a5c4d-3e2f-1a0b-9c8d-7e6f5a4b3c2d
  blockUid: cc112233-4455-6677-8899-aabbccddeeff
  originalFile: /payloads/create-user.void
---
```

### `importedFrom` Attribute

Blocks that were pulled in via a `linkedBlock` carry `importedFrom` on their attrs (e.g. `importedFrom: "/shared/base-headers.void"`). This is set **automatically by Voiden** when resolving links — never set it manually when generating `.void` files.

## File Naming

- kebab-case: `list-users.void`, `get-user.void`, `create-order.void`
- Group by resource in folders: `users/`, `orders/`, `auth/`
- One logical resource group per file; multiple request blocks per file is fine

## Environment Variables

Store secrets in `.env` alongside `.void` files:

```env
BASE_URL=https://api.example.com
API_TOKEN=your-token-here
USER_ID=123
```

Reference with `{{VARIABLE_NAME}}` in any block field:
- URL: `{{BASE_URL}}/users/{{USER_ID}}`
- Header value: `Bearer {{API_TOKEN}}`
- Body field: `"email": "{{USER_EMAIL}}"`
